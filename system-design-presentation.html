Detailed System Architecture

Architecture Decision: Separate Dashboards

YES - We will create TWO separate dashboard applications:

  • Super Admin Dashboard: Your internal management tool (admin.yourapp.com)
  • School Admin Dashboard: Client-facing portal (app.yourapp.com or custom domains)

Why? Complete separation of concerns, different user experiences, easier to maintain, and allows white-labeling for schools.

Multi-Tenancy Key Concept

Notice the school_id column in every table? This ensures complete data isolation. When a school admin queries students, the backend automatically filters: WHERE school_id = {their_school_id}

Security & Best Practices

Authentication Security

  • Password Hashing: Use bcrypt with cost factor 12
  • JWT Tokens: Short expiry (15 min) + refresh tokens
  • HTTPS Only: Enforce SSL/TLS in production
  • Rate Limiting: Prevent brute force attacks (5 attempts/5 min)
  • 2FA Option: For super admin accounts

Data Security

  • Input Validation: Sanitize all user inputs (prevent SQL injection)
  • Parameterized Queries: Use prepared statements
  • Encryption at Rest: Sensitive fields encrypted in DB
  • Audit Logging: Track all data access/modifications
  • Backup Strategy: Daily backups with 30-day retention

Multi-Tenancy Security

  • Row-Level Security: EVERY query filters by school_id
  • Middleware Check: Verify school_id in JWT matches query
  • API Key Scope: Devices can only log for their school
  • No Cross-School Access: School A can NEVER see School B data
  • Test Isolation: Penetration test multi-tenancy

Compliance

  • FERPA (US): Student privacy in educational records
  • GDPR (EU): If serving international schools
  • COPPA: Children's online privacy (under 13)
  • Data Retention: Clear policy (e.g., 7 years)
  • Right to Deletion: Parents can request data removal

Critical Security Checklist

Never Store:
  • Plain text passwords
  • API keys in code
  • Secrets in Git
Always Use:
  • Environment variables
  • HTTPS in production
  • Helmet.js for headers
Always Validate:
  • User inputs
  • File uploads
  • JWT tokens
Always Log:
  • Failed login attempts
  • Data access
  • System errors

Complete Technical Flow

Request Flow Example: School Admin Views Dashboard

1. School Admin

Opens dashboard in browser

2. React Frontend

Sends GET request to /api/v1/school/dashboard/today
Includes JWT token in Authorization header

3. Auth Middleware

Verifies JWT token
Extracts school_id = 5, role = 'school_admin'

4. Multi-Tenant Middleware

Attaches school_id to request object
req.schoolId = 5

5. Controller Logic

Calls database queries with school_id filter

6. PostgreSQL Query

SELECT COUNT(*) FROM attendance_logs
WHERE school_id = 5 AND date = TODAY
GROUP BY status

7. API Response

JSON: { present: 450, absent: 25, late: 12 }

8. React Updates UI

Displays stat cards with numbers
