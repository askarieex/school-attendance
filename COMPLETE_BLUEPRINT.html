System Architecture

Critical Decision: TWO Separate Dashboards

YES - We will create TWO separate dashboard applications:

  • Super Admin Dashboard (admin.attendanceapp.com) - Your internal management tool
  • School Admin Dashboard (app.attendanceapp.com) - Client-facing portal

Why? Complete separation of concerns, different UX, easier maintenance, white-labeling capability.

6-Layer Architecture

Presentation Layer (Frontend)

Super Admin Dashboard: React SPA, dark theme, comprehensive controls

School Dashboard: React SPA, light theme, simplified interface

Parent Mobile App: React Native (future phase)

API Gateway Layer

Technology: Express.js with middleware pipeline

Responsibilities: Request routing, authentication, rate limiting, logging

Endpoints: RESTful API with versioning (v1, v2)

Business Logic Layer

Controllers: Handle HTTP requests/responses

Services: Core business logic (attendance, notifications, reports)

Validators: Input validation and sanitization

Data Access Layer

ORM: Prisma or Sequelize for database operations

Models: School, User, Student, Attendance, Device, Settings

Repositories: Abstraction over database queries

Database Layer

Primary: PostgreSQL 14+ (relational data)

Cache: Redis (session storage, real-time data)

Storage: AWS S3 (student photos, reports)

Integration Layer

SMS: Twilio API for parent notifications

Email: SendGrid or AWS SES

Hardware: Webhook receiver for RFID devices

Payments: Stripe for subscription billing

Database Schema

Multi-Tenancy Pattern

Core Rule: Every table has school_id. Every query MUST filter by school_id for complete data isolation.

6 Core Tables

schools
id (SERIAL PRIMARY KEY) name (VARCHAR 255) email (VARCHAR 255 UNIQUE) phone (VARCHAR 20) address (TEXT) is_active (BOOLEAN DEFAULT TRUE) created_at (TIMESTAMP)
users
id (SERIAL PRIMARY KEY) email (VARCHAR 255 UNIQUE) password_hash (VARCHAR 255) role (ENUM: 'superadmin', 'school_admin') school_id (INT FK → schools) full_name (VARCHAR 255) created_at (TIMESTAMP)
devices
id (SERIAL PRIMARY KEY) api_key (VARCHAR 255 UNIQUE) device_name (VARCHAR 255) school_id (INT FK → schools) is_active (BOOLEAN DEFAULT TRUE) last_seen (TIMESTAMP) created_at (TIMESTAMP)
students
id (SERIAL PRIMARY KEY) full_name (VARCHAR 255) grade (VARCHAR 50) rfid_card_id (VARCHAR 100 UNIQUE) parent_name (VARCHAR 255) parent_phone (VARCHAR 20) parent_email (VARCHAR 255) school_id (INT FK → schools) is_active (BOOLEAN DEFAULT TRUE) created_at (TIMESTAMP)
attendance_logs
id (SERIAL PRIMARY KEY) student_id (INT FK → students) school_id (INT FK → schools) check_in_time (TIMESTAMP) status (ENUM: 'present', 'late', 'absent') device_id (INT FK → devices) date (DATE)
school_settings
school_id (INT PRIMARY KEY FK) school_start_time (TIME) late_threshold_minutes (INT DEFAULT 15) sms_enabled (BOOLEAN DEFAULT TRUE) timezone (VARCHAR 50)

Example Queries

-- School Admin (school_id = 12) views their students SELECT * FROM students WHERE school_id = 12 -- Multi-tenant filter AND is_active = true; -- Get today's attendance for school SELECT status, COUNT(*) as count FROM attendance_logs WHERE school_id = 12 AND date = CURRENT_DATE GROUP BY status;
